# 품질 규칙

## TDD 규칙

1. **테스트 먼저**: 모든 기능은 실패하는 테스트를 먼저 작성한다
2. **최소 구현**: 테스트를 통과하는 최소한의 코드만 작성한다
3. **리팩토링 안전망**: 리팩토링 후 기존 테스트가 모두 통과하는지 확인한다
4. **커버리지 목표**: 핵심 로직 80% 이상

## 점진적 하네스 규칙

1. 에이전트가 동일한 실수를 **2회 이상 반복**하면, `CLAUDE.md` 또는 이 문서에 해당 규칙을 추가한다
2. 문서 규칙으로도 위반이 반복되면, **린터/테스트/CI로 승격**하여 기계적으로 강제한다
3. 하네스는 "필요할 때 추가"하는 것이지, 미리 완벽하게 설계하는 것이 아니다
4. 규칙 추가 시 **왜 추가했는지 (어떤 실패가 반복되었는지)** 주석으로 기록한다

## 아키텍처 제약

1. **의존성 방향 엄수**: 아키텍처 문서에 정의된 의존성 방향을 따른다. 역방향 금지
2. **경계 검증**: 모든 외부 데이터(API 응답, 파일 파싱, 사용자 입력)는 시스템 경계에서 검증
   <!-- 스택별 힌트: TypeScript → Zod, Python → Pydantic, Go → validator 등 -->
3. **파일 크기 제한**: 파일당 최대 300줄 (초과 시 분할)
4. **함수 크기 제한**: 함수당 최대 50줄

## 코딩 규칙

1. **타입 안전성**: 느슨한 타입 사용 금지. 명시적 타입 선언 필수
   <!-- 스택별 힌트: TypeScript → `any` 금지, Python → type hints 필수, Go → `interface{}` 최소화 -->
2. **디버그 출력 금지**: 구조화된 로거만 사용. stdout/stderr 직접 출력 금지
   <!-- 스택별 힌트: TypeScript → console.log 금지, Python → print() 금지 -->
3. **매직 넘버 금지**: 상수로 추출하여 이름 부여
4. **모듈 시스템**: 프로젝트에서 채택한 모듈 시스템의 규칙을 따른다
   <!-- 스택별 힌트: TypeScript ESM → .js 확장자, Python → __init__.py, Go → go.mod -->
5. **에러 처리**: throw/raise 대신 에러를 값으로 반환하는 패턴 권장. 불가피한 경우 명시적 에러 타입 사용
   <!-- 스택별 힌트: TypeScript → Result 타입, Go → (value, error) 튜플, Rust → Result<T, E> -->

## 테스트 규칙

1. **테스트 파일 위치**: 테스트 디렉토리에 소스 구조 미러링
2. **픽스처**: 테스트 픽스처 디렉토리에 보관. 테스트 코드 내에 대량의 테스트 데이터를 인라인하지 않음
3. **테스트 이름**: 동작을 설명하는 자연어 형식. 한국어 또는 영어 모두 허용
4. **모킹 최소화**: 실제 로직 테스트 우선. 외부 서비스만 모킹
