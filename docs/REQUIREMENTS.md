# 요구사항 (EARS 기반)

EARS(Easy Approach to Requirements Syntax) 패턴으로 작성된 요구사항.
MECE 도메인별로 분류.

## EARS 패턴 참조

| 패턴 | 키워드 | 템플릿 |
|------|--------|--------|
| Ubiquitous | (없음) | "시스템은 {응답}해야 한다" |
| State-driven | While | "While {조건}, 시스템은 {응답}해야 한다" |
| Event-driven | When | "When {트리거}, 시스템은 {응답}해야 한다" |
| Unwanted | If...Then | "If {조건}, Then 시스템은 {응답}해야 한다" |
| Optional | Where | "Where {기능 포함 시}, 시스템은 {응답}해야 한다" |
| Complex | While+When | "While {조건}, When {트리거}, 시스템은 {응답}해야 한다" |

## MECE 도메인 분해 원칙

- 도메인은 **MECE**(Mutually Exclusive, Collectively Exhaustive)로 분해한다
- 각 도메인은 독립적이고 겹치지 않으며, 전체를 빠짐없이 커버한다
- 도메인 코드는 `D{번호}` 형식 (예: D1, D2, D3)
- 각 도메인 내에서 하위 도메인은 `D{번호}.{하위번호}` 형식
- 요구사항 ID는 `[D{n}.{m}-{순번}]` 형식 (예: [D1.1-01])

---

## D1: Badge Detection (뱃지 감지)

### D1.1: 파란 뱃지 식별

- **[D1.1-01][When]** When 트윗 또는 프로필이 렌더링되면, 시스템은 파란 뱃지(인증 마크) 존재 여부를 감지해야 한다
- **[D1.1-02][When]** When 파란 뱃지가 감지되면, 시스템은 해당 계정이 수익성 목적 뱃지인지 판별해야 한다

## D2: Content Filtering (콘텐츠 필터링)

### D2.1: 콘텐츠 숨김

- **[D2.1-01][While]** While 익스텐션이 활성화된 상태에서, 수익성 뱃지 계정의 트윗은 숨겨야 한다
- **[D2.1-02][While]** While 익스텐션이 활성화된 상태에서, 예외 목록(팔로우/화이트리스트)에 포함된 계정은 숨기지 않아야 한다
- **[D2.1-03][When]** When X 페이지에서 새로운 트윗이 동적으로 로드되면, 시스템은 새 트윗에도 필터링을 적용해야 한다

## D3: Follow & Whitelist (팔로우 & 화이트리스트)

### D3.1: 팔로우 목록 조회

- **[D3.1-01][When]** When 사용자가 팔로우 동기화를 요청하면, 시스템은 X 토큰을 사용하여 팔로우 목록을 조회해야 한다
- **[D3.1-02][Ubiquitous]** 조회된 팔로우 목록은 Chrome Storage에 캐시해야 한다

### D3.2: 수동 화이트리스트

- **[D3.2-01][When]** When 사용자가 특정 계정을 화이트리스트에 추가하면, 시스템은 해당 계정을 예외 목록에 저장해야 한다
- **[D3.2-02][When]** When 사용자가 화이트리스트에서 계정을 제거하면, 시스템은 해당 계정을 예외 목록에서 삭제해야 한다

## D4: Settings (설정)

### D4.1: 사용자 설정

- **[D4.1-01][When]** When 사용자가 익스텐션 아이콘을 클릭하면, 시스템은 Popup UI를 표시해야 한다
- **[D4.1-02][When]** When 사용자가 on/off 토글을 변경하면, 시스템은 필터링 상태를 즉시 반영해야 한다
- **[D4.1-03][Ubiquitous]** 사용자 설정은 Chrome Storage에 영속적으로 저장해야 한다

## D5: Keyword Filter (키워드 필터)

### D5.1: 키워드 기반 선별 필터링

- **[D5.1-01][While]** While 키워드 필터가 활성화된 상태에서, 파딱 계정 중 키워드 규칙에 매칭되는 계정만 숨겨야 한다
- **[D5.1-02][While]** While 키워드 필터가 활성화된 상태에서, 키워드에 매칭되지 않는 파딱 계정은 표시해야 한다
- **[D5.1-03][When]** When 키워드 규칙이 변경되면, 시스템은 기존 트윗을 새 규칙으로 재처리해야 한다

### D5.2: 키워드 규칙 관리

- **[D5.2-01][Ubiquitous]** 키워드 규칙은 한 줄에 하나씩 작성하며, `*` 와일드카드와 `@@handle` 예외를 지원해야 한다
- **[D5.2-02][Ubiquitous]** 내장 필터는 카테고리별로 분류되며, 각 카테고리를 개별 토글할 수 있어야 한다
- **[D5.2-03][When]** When 사용자가 커스텀 필터를 저장하면, 시스템은 내장 필터와 합쳐 적용해야 한다

### D5.3: 프로필 캐시

- **[D5.3-01][Ubiquitous]** 키워드 매칭 대상(handle, displayName, bio, tweetText)은 프로필 캐시에서 조회해야 한다
- **[D5.3-02][When]** When 프로필 데이터가 뒤늦게 도착하면, 시스템은 해당 계정의 트윗을 재처리해야 한다
